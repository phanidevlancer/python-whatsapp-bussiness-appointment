# WhatsApp Appointment Booking

A production-ready WhatsApp chatbot that automates appointment booking using the official **Meta WhatsApp Cloud API** — no third-party providers.

---

## Architecture

```
User (WhatsApp)
      │
      │  HTTPS
      ▼
Meta WhatsApp Cloud API
      │
      │  POST /webhook (inbound messages)
      │  GET  /webhook (verification)
      ▼
FastAPI  (main.py)
      │
      ├── routers/webhook.py          ← receives & dispatches
      │
      ├── services/booking_flow_service.py  ← state machine
      │       │
      │       ├── repositories/       ← async DB queries
      │       ├── services/session_service.py  ← Redis (idempotency + slot locks)
      │       └── integrations/whatsapp_client.py  ← outbound API calls
      │
      ├── PostgreSQL  (SQLAlchemy + asyncpg)
      └── Redis       (idempotency keys + slot locks)
```

### Conversation flow

```
User: "hi"
  → Bot sends service list (interactive list message)

User: selects "General Consultation"
  → Bot stores selection, sends available time slots

User: selects "Mar 28, 10:00 AM"
  → Redis lock acquired, Bot sends confirmation buttons

User: taps "Confirm"
  → DB transaction: slot locked (SELECT FOR UPDATE NOWAIT) + appointment created
  → Bot sends booking confirmation with reference number
```

## Setup

### 1. Prerequisites

- Python 3.12+
- PostgreSQL running locally (or remote)
- Redis running locally (or remote)
- A Meta Developer account with a WhatsApp Business App

### 2. Create virtual environment

```bash
python3.12 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
```

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

### 4. Configure environment

```bash
cp .env.example .env
```

Edit `.env` and fill in:

```
DATABASE_URL=postgresql+asyncpg://postgres:yourpassword@localhost:5432/whatsapp_booking
REDIS_URL=redis://localhost:6379/0
WHATSAPP_TOKEN=<your Meta permanent access token>
WHATSAPP_PHONE_NUMBER_ID=<your phone number ID from Meta dashboard>
WHATSAPP_VERIFY_TOKEN=<any random string you choose, used during webhook setup>
```

### 5. Create the database

```bash
psql -U postgres -c "CREATE DATABASE whatsapp_booking;"
```

## Database Migrations

### Initialize Alembic (first time only)

Alembic is already configured. Just run migrations:

```bash
# Create the initial migration (autogenerated from your models)
alembic revision --autogenerate -m "initial schema"

# Apply migrations to the database
alembic upgrade head
```

### Future migrations

Whenever you change a model:

```bash
alembic revision --autogenerate -m "describe your change"
alembic upgrade head
```

For the RBAC rollout, run the migration first and seed second:

```bash
alembic upgrade head
python -m app.db.seed
```

The RBAC seed is idempotent. It creates the permission catalog, the default role templates, and backfills existing admin users onto matching templates.

### Useful Alembic commands

```bash
alembic current          # Show current revision
alembic history          # Show migration history
alembic downgrade -1     # Roll back one migration
```

## RBAC Bootstrap

Run this after migrations to create the default permission catalog and system role templates:

```bash
python -m app.db.seed
```

Default system templates:
- `Super Admin`
- `Admin`
- `Receptionist`
- `Viewer`
- `Operations Manager`

First-login flow:
- New admin users are created with `must_change_password=True` and `is_first_login=True`.
- The frontend should redirect those users to the change-password screen on first login.
- After the password is changed, the old token is invalidated and a fresh session is issued.

Optional startup bootstrap for local development only:

```bash
RBAC_BOOTSTRAP_ON_STARTUP=true uvicorn main:app --reload
```

This is guarded in `main.py` so it only runs in non-production environments.

## Webhook Setup (Local Development)

Meta requires a publicly accessible HTTPS URL for your webhook.

### Option A: ngrok

```bash
# Install: https://ngrok.com/download
ngrok http 8000
# Note the https URL: e.g., https://abc123.ngrok.io
```

### Option B: Cloudflare Tunnel

```bash
cloudflared tunnel --url http://localhost:8000
```

### Register with Meta

1. Go to [Meta Developer Console](https://developers.facebook.com)
2. Navigate to your app → WhatsApp → Configuration
3. Set **Webhook URL** to: `https://your-tunnel-url/webhook`
4. Set **Verify Token** to the same value as `WHATSAPP_VERIFY_TOKEN` in your `.env`
5. Click **Verify and Save**
6. Subscribe to the **messages** field under Webhook Fields

### Test the verification

```bash
curl "http://localhost:8000/webhook?hub.mode=subscribe&hub.verify_token=YOUR_VERIFY_TOKEN&hub.challenge=test123"
# Should return: test123
```

## Slot Conflict Prevention

Double-booking is prevented at two layers:

1. **Redis `SET NX`** (fast): When a user selects a slot, a Redis key with a 5-minute TTL is set atomically. If another user tries to lock the same slot, they get an immediate "slot taken" response — no DB hit needed.

2. **`SELECT FOR UPDATE NOWAIT`** (guaranteed): When the user confirms, the slot row is locked at the DB level. `NOWAIT` means it raises immediately if another transaction holds the lock (no deadlock waiting). Combined with a `UNIQUE` constraint on `appointments.slot_id`, double-booking is impossible even if two requests race past the Redis layer simultaneously.

---

## Idempotency

Meta may deliver the same webhook event more than once (retries on non-2xx or timeout). Each message has a unique `message_id`. Before processing, `session_service.is_duplicate_message()` checks a Redis key for that ID. If it exists, the message is skipped. The key expires after 24 hours.
